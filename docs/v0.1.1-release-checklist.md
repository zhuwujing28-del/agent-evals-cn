# v0.1.1 Release Checklist

Use this checklist for the first patch tag after the published `v0.1.0` baseline.

## Scope

`v0.1.1` should package the eval-readiness hardening that already landed on
`main` after the older `v0.1.0` tag. It should not add new baseline cases during
the tagging pass.

Included readiness changes:

- eval PR template,
- security redaction and unsafe-eval boundary policy,
- context budget eval case,
- MCP vs Codex skill boundary eval case,
- scoring rubric,
- CI readiness badge,
- release checklist,
- baseline eval case ID and required-section validation,
- prepared `v0.1.0` changelog.

## Preconditions

- `python scripts/validate.py` passes locally.
- The public `validate` workflow is green on the exact commit to tag.
- `CHANGELOG.md` has a `v0.1.1` entry summarizing only post-`v0.1.0` eval
  hardening.
- `docs/post-v0.1.0-audit-2026-07-13.md` still matches the public tag target.
- Examples and usage reports contain no private logs, customer data, access
  tokens, or unpublished repository details.

## Release steps

1. Review baseline cases under `examples/` for placeholder drift.
2. Run `python scripts/validate.py`.
3. Confirm `git status --short --branch` is clean and aligned with `origin/main`.
4. Confirm the GitHub Actions badge points to a successful `validate` run.
5. Move the relevant `Unreleased` changelog notes under `v0.1.1`.
6. Create an annotated tag:

   ```powershell
   git tag -a v0.1.1 -m "agent-evals-cn v0.1.1"
   git push origin v0.1.1
   ```

7. Check the public tag page renders README links, eval cases, and docs links.

## Do not release if

- Any validation check fails locally or in GitHub Actions.
- A baseline eval case cannot be judged by a maintainer using the scoring
  rubric.
- The changelog claims automated scoring that the project does not provide.
- The checklist is being used to rewrite or move the existing `v0.1.0` tag.
