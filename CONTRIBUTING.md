# 贡献指南

欢迎提交新的 eval case、评测报告模板、维护文档和结构校验改进。这个仓库关注的是可复用、可回归、可人工复核的 Codex / agent 工作流评测，不收录泛泛的提示词集合。

## 提交前请确认

- 你的变更对应一个真实的维护或协作场景，而不是抽象能力描述。
- 新增 eval case 放在 `examples/*-case.md`，并包含唯一的 `## case-id:`。
- 新增 eval case 已同步更新 `docs/eval-case-index.md`。
- case 中不包含真实密钥、私有日志、未脱敏漏洞细节或账号数据。
- 期望行为和失败判定足够具体，维护者可以独立复核。

## 推荐流程

1. 先阅读 `examples/eval-case-template.md` 和 `docs/scoring-rubric.md`。
2. 新增或修改一个最小可复核的 eval case。
3. 如果新增 case，更新 `docs/eval-case-index.md` 的能力和失败模式。
4. 运行：

   ```powershell
   python .\scripts\validate.py
   ```

5. 在 PR 中说明目标能力、主要失败模式、验证结果和任何安全边界。

## 可接受的贡献

- 从真实失败中提炼出的中文维护场景 eval case。
- 能减少人工评审分歧的评分标准或复核流程。
- 能让安装、运行、发布或验证更清楚的文档。
- 小而明确的校验脚本改进。

## 不接受的贡献

- 无法复现或无法判定成败的宽泛 prompt。
- 鼓励绕过访问控制、登录限制、付费墙或站点规则的网页提取案例。
- 泄露私有仓库、客户数据、漏洞细节或密钥的样例。
- 大批量生成但没有明确失败模式的 case。

## PR 说明建议

请尽量包含：

- 这次变更覆盖的能力。
- 它要防止的主要回归或失败模式。
- 是否来自真实失败，以及是否已脱敏。
- 已运行的验证命令。
- 后续还需要人工复核的风险点。
